# Обнуление остатков на Ozon для карточек, отсутствующих в базе

**Дата:** 2026-07-12
**Статус:** утверждено

## Проблема

Задача `update_remains_ozon` (`ozon/tasks.py`) отправляет остатки на Ozon только по товарам,
у которых есть запись `OzonData` в локальной базе. Карточки, существующие на Ozon, но
отсутствующие в базе (нет записи `OzonData` / нет соответствия по `offer_id`), остаток
не получают — они продолжают продаваться со старым (устаревшим) остатком. Это приводит
к заказам на товары, которых фактически нет.

Примечание: `Product.stock` — `PositiveIntegerField(default=0)`, NOT NULL, поэтому
для товаров из базы `"stock": null` в payload невозможен — отдельный фикс не нужен.

## Решение

После штатной отправки остатков в рамках той же Celery-задачи `update_remains_ozon`:

1. Получить с Ozon полный список карточек через `POST /v3/product/list`
   (пагинация по `last_id`, `limit=1000`, `filter.visibility="ALL"`).
2. Пропустить карточки с `archived=true` (архивные не продаются).
3. Вычислить разницу: `offer_id` с Ozon, отсутствующие среди `offer_id` в `OzonData`.
4. По разнице отправить `stock: 0` через `POST /v2/products/stocks`
   (батчи по 100 позиций, `warehouse_id=22865154657000` — тот же, что в текущем коде).

Порядок гарантирован: сначала свои остатки, потом обнуление неизвестных — обе фазы
внутри одной задачи.

## Факты из Ozon Seller API

- `POST /v2/products/stocks`: до 100 пар товар-склад за запрос; лимит 80 запросов/мин
  на аккаунт; достаточно `offer_id` + `stock` + `warehouse_id` (при передаче и
  `offer_id`, и `product_id` приоритет у `offer_id`). Ошибки по позициям приходят в
  `result[].errors` (например `offer_id_not_found`, `product_is_not_created`) — они
  не валят весь запрос.
- `POST /v3/product/list`: возвращает все карточки продавца парами
  `offer_id`/`product_id` + флаг `archived`; пагинация: первый запрос с пустым
  `last_id`, далее `last_id` из ответа; `limit` 1—1000. Конец — когда `items` пуст
  либо `last_id` пустой.

## Компоненты

### `OzonExchange.get_product_list(last_id="", limit=1000)` — новый метод

`src/lekala_class/class_marketplace/OzonExchange.py`. Один вызов `/v3/product/list`
с `filter={"visibility": "ALL"}`, возвращает сырой ответ. Пагинацию крутит вызывающий код.

### `close_unknown_ozon_stocks(dry_run=False)` — новая функция в `ozon/tasks.py`

- Собирает все неархивные `offer_id` с Ozon (цикл по страницам `get_product_list`).
- `known = set(OzonData.objects.values_list('offer_id', flat=True))`.
- `unknown = ozon_offer_ids - known`.
- При `dry_run=True`: только лог списка и количества, ничего не отправляет.
- Иначе: батчи по 100 → `ozon_api.update_remains(data=[{"offer_id": ..., "stock": 0,
  "warehouse_id": 22865154657000, "quant_size": 1}, ...])`. Между запросами пауза
  `time.sleep(1)` (страховка от лимита 80 req/min; батчей обычно единицы).
- Логирует: всего карточек на Ozon, известных в базе, обнулено, позиции с ошибками
  из `result[].errors`.
- Возвращает список обнулённых `offer_id` (для тестов и dry-run-отчёта).

### Изменение `update_remains_ozon`

- В конце: вызов `close_unknown_ozon_stocks()` в `try/except Exception` с логом —
  сбой сверки не должен ронять основную отправку остатков.

## Обработка ошибок

- Ошибка любой страницы `/v3/product/list` → прервать фазу сверки с логом и ничего
  не отправлять в этом прогоне. Неполный список сам по себе безопасен (обнулили бы
  просто меньше карточек), но при ошибке ответ может быть некорректным — надёжнее
  дождаться следующего прогона.
- Ошибки по позициям в ответе `/v2/products/stocks` — залогировать, продолжить.
- Вся фаза сверки обёрнута в try/except в `update_remains_ozon`.

## Безопасность / порядок ввода в прод

- В dev база неактуальна → в dev запускать только `close_unknown_ozon_stocks(dry_run=True)`.
- Первый запуск в проде — вручную с `dry_run=True`, просмотреть список offer_id,
  затем включить боевой режим (вызов без параметра из `update_remains_ozon`).
- Механизм обратим: появление товара в базе (запись `OzonData`) вернёт остаток
  штатным путём при следующем прогоне.

## Тесты (`ozon/tests.py`, мок `OzonExchange`)

1. Пагинация: две страницы `/v3/product/list` склеиваются, `archived=true` пропускается.
2. Разница множеств: обнуляются только offer_id, отсутствующие в `OzonData`.
3. Батчинг: 250 неизвестных → 3 вызова `update_remains` (100/100/50), везде `stock=0`.
4. `dry_run=True`: `update_remains` не вызывается, список возвращается.
5. Ошибка страницы списка: фаза прекращается, `update_remains` не вызывается.
6. Исключение внутри `close_unknown_ozon_stocks` не роняет `update_remains_ozon`.

## Вне скоупа

- Архивация карточек на Ozon.
- Синхронизация/создание записей `OzonData` для неизвестных карточек.
- Изменение расписания задач (задача по-прежнему дёргается из `after_catalog_update`).
