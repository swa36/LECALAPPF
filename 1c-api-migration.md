# Миграция API обмена 1С ↔ сайт: старое ТЗ → новый сервис

Документ для разработчика сайта. Показывает разницу между **исходным ТЗ**
(`1c-api-tz.md` — прямые OData-запросы к типовой 1С) и **реализованным API**
(кастомный HTTP-сервис `prokopov` в расширении `WEB_API_Prokopov`).

Цель — чтобы разработчик сайта мог переписать клиент под новые эндпоинты.

> Статус: API реализован и проверен на живой опубликованной базе (все эндпоинты
> отвечают `200`, POST создаёт и проводит документ `ЗаказКлиента`).

---

## 0. Что изменилось концептуально (читать первым)

| Было (ТЗ) | Стало (реализация) |
|---|---|
| Прямой OData-доступ (`/odata/standard.odata/...`) | Единый HTTP-сервис с «человеческими» путями `/hs/prokopov/...` |
| Имена полей 1С: `Ref_Key`, `Парент_Key`, кириллица (`Артикул`, `Цена`) | Все ключи ответов в `snake_case` (`ref_key`, `article`, `price`) |
| Отдельные запросы: товары, цены, остатки, изображения | **Цены и остатки встроены в `products`**. Изображения — отдельный эндпоинт по UUID |
| Цены/остатки — сырые движения регистров, сайт сам считает | 1С отдаёт **готовые** значения (срез последних цен, рассчитанный остаток) |
| Состояние заказа + состав — 2 запроса (регистр + документ) | Объединено в один `orders-in-shipping` |
| Партнёр / контрагент / создание / проведение заказа — 4 POST | **Один POST `/order`** делает всё внутри транзакции |
| Ответы — обычный JSON | Ответы **сжаты gzip** (нужен `Accept-Encoding: gzip` / `curl --compressed`) |

**Кол-во вызовов на обновление каталога:** было ~6 (+ N запросов изображений),
стало **5** (`products` уже несёт цены и остатки; изображения — по требованию).

## 1. Общие требования

### 1.1. Базовый URL

```
http://<host>/<имя_базы>/hs/prokopov
```

Пример (тестовая публикация): `http://localhost/YT_Prokopov/hs/prokopov`

Корневой URL сервиса — `prokopov` (свойство HTTP-сервиса `wapi_Обмен`).

### 1.2. Авторизация

Basic Auth — без изменений относительно ТЗ.

### 1.3. Сжатие ответов (ВАЖНО, новое)

Все ответы отдаются в **gzip**. Клиент обязан:
- слать заголовок `Accept-Encoding: gzip`;
- распаковывать тело.

В `requests` (Python) распаковка идёт автоматически. В `curl` — флаг `--compressed`.
Без распаковки сайт получит бинарный мусор.

### 1.4. Кодировка тела POST (ВАЖНО, новое)

Тело POST `/order` использует **кириллические ключи** (`Покупатель`, `Товары`...).
Тело обязательно слать в **UTF-8** с заголовком
`Content-Type: application/json; charset=utf-8`.
При другой кодировке 1С не разберёт JSON и вернёт `500`.

### 1.5. Формат ошибок (новое, единый)

Любая ошибка приходит в виде:

```json
{ "error": "текст ошибки", "code": "MACHINE_CODE" }
```

| HTTP | code | Когда |
|---|---|---|
| 400 | `INVALID_UUID` | Некорректный UUID в пути (изображения) |
| 400 | `MISSING_ITEMS` | В заказе нет товаров |
| 400 | `MISSING_BUYER` | В заказе не указан покупатель |
| 404 | `ITEM_NOT_FOUND` | Товар заказа не найден по UUID (в тексте — номер строки) |
| 404 | `UNKNOWN_ENDPOINT` | Неизвестный путь |
| 500 | `INTERNAL_ERROR` | Внутренняя ошибка 1С (текст в `error`) |

## 3. GET-эндпоинты: детальный маппинг

### 3.1. `GET /products` — товары (ТЗ 2.1 + 2.5 + 2.6)

Постраничный список товаров. **Цены и остаток встроены в каждый товар.**

**Параметры запроса:**

| Параметр | По умолчанию | Огранич. | Назначение |
|---|---|---|---|
| `page` | 1 | ≥ 1 | Номер страницы |
| `page_size` | 200 | 1…500 | Размер страницы |

**Пример:** `GET /products?page=1&page_size=10`

**Маппинг полей товара (старое → новое):**

| Было (ТЗ 2.1) | Стало | Тип | Комментарий |
|---|---|---|---|
| `Ref_Key` | `ref_key` | string(uuid) | UUID товара |
| `Parent_Key` | `parent_key` | string(uuid) | UUID категории |
| `DataVersion` | `data_version` | string | Версия объекта 1С |
| `IsFolder` | `is_folder` | bool | Всегда `false` (товары) |
| `Code` | `code` | string | Код 1С |
| `Артикул` | `article` | string | Артикул |
| `ВесЧислитель` | `weight_numerator` | number | Вес |
| `Description` | `description` | string | Наименование |
| `Описание` | `description_text` | string | Описание (текст) |
| `ФайлКартинки_Key` | `picture_file_key` | string(uuid) | UUID главного изображения |
| `ДополнительныеРеквизиты[]` | `additional_attributes[]` | array | Характеристики (см. ниже) |
| `ДополнительныеРеквизиты[].Свойство_Key` | `additional_attributes[].property_key` | string(uuid) | UUID характеристики |
| `ДополнительныеРеквизиты[].Значение` | `additional_attributes[].value` | string | Значение |
| — (новое) | `additional_attributes[].description` | string | Название характеристики (бонус, можно игнорировать) |

**Встроенные цены (заменяют ТЗ 2.5):**

**Встроенный остаток (заменяет ТЗ 2.6):**

| Поле | Тип | Комментарий |
|---|---|---|
| `in_stock` | number | Готовый остаток «в наличии» на текущую дату |

**Поля пагинации (новое, оболочка ответа):**

| Поле | Тип | Комментарий |

| `page` | number | Текущая страница |
| `page_size` | number | Размер страницы |
| `pages` | number | Всего страниц |

### 3.5. `GET /orders-in-shipping` — заказы в отгрузке (ТЗ 2.7 + 2.8 объединены)

Раньше было два запроса: получить состояния (регистр), затем по каждому заказу
запросить документ. Теперь **один запрос** возвращает заказы в статусе
«в процессе отгрузки» сразу с составом товаров.

| Было (ТЗ 2.7/2.8) | Стало | Тип |
|---|---|---|
| `Заказ` (UUID заказа) | `order_key` | string(uuid) |
| `Товары[]` (состав) | `items[]` | array |
| `Товары[].Номенклатура_Key` | `items[].nomenclature_key` | string(uuid) |
| `Товары[].Количество` | `items[].quantity` | number |

Оболочка — `{ "value": [ ... ] }`.

```json
{
  "value": [
    {
      "order_key": "96d6b83a-4800-11f1-8001-00155d8b1954",
      "items": [
        { "nomenclature_key": "127c775a-5265-11f0-9d1b-00155d46f78d", "quantity": 1 }
      ]
    }
  ]
}
```

**Что изменилось в логике сайта:**
- ❌ Убрать запрос состояний и последующие запросы документов.
- ✅ Фильтр по статусу «в процессе отгрузки» делает 1С — сайт получает только нужные заказы.
- Поле `Состояние` в ответе не нужно (всё уже отфильтровано), `order_key` = бывший `Ref_Key` заказа.

`{uuid}` в пути — UUID товара (`products[].ref_key`).

| — (новое) | `images[].is_main` | bool — главное изображение (= `picture_file_key` товара) |
| — (новое) | `nomenclature_key` | string(uuid) — эхо UUID товара |

```json
{
  "nomenclature_key": "7e019266-24a4-11ef-8009-00155d46f78d",
  "images": [
    { "file_uuid": "f5a159fc-...", "is_main": true, "data_base64": "/9j/4AAQSkZJRgABAQ..." }
  ]
}
```

**Что изменилось в логике сайта:**
- ❌ Убрать различение старого/нового хранилища — 1С сама достаёт содержимое.
- ✅ Главное изображение помечено `is_main: true` (можно не сверять с `picture_file_key`).
- При некорректном UUID — `400 INVALID_UUID`.

## 4. POST `/order` — создание заказа (ТЗ 3.1 + 3.2 + 3.3 + 3.4 объединены)

Раньше: 4 запроса (создать партнёра → контрагента → заказ → провести).
Теперь **один POST** делает всё внутри транзакции 1С: ищет/создаёт партнёра и
контрагента по имени покупателя, создаёт и **сразу проводит** документ.

**ВХОДНЫЕ ключи — кириллица (НЕ переименованы, оставлены для совместимости).**
Тело — строго UTF-8.

**Тело запроса:**

| Поле (вход) | Тип | Обязат. | Что отправлять |
|---|---|---|---|
| `Покупатель` | string | да | Имя покупателя / строка маркетплейса. По нему ищется/создаётся партнёр и контрагент |
| `Номер` | string | нет | Номер заказа (формат маркетплейса, напр. `OZ00-123456789`). Если пусто — 1С присвоит свой |
| `Комментарий` | string | нет | Комментарий (маркетплейс + номер) |
| `СуммаДокумента` | number | нет | Сумма заказа |
| `Товары[]` | array | да (непустой) | Строки заказа |
| `Товары[].Номенклатура_Key` | string(uuid) | да | UUID товара в 1С |
| `Товары[].Количество` | number | да | Количество |
| `Товары[].Цена` | number | да | Цена |

**Пример тела:**

```json
{
  "Покупатель": "OZON 123456789",
  "Номер": "OZ00-123456789",
  "Комментарий": "OZON\n123456789",
  "СуммаДокумента": 1991,
  "Товары": [
    { "Номенклатура_Key": "7e019266-24a4-11ef-8009-00155d46f78d", "Количество": 1, "Цена": 1991 }
  ]
}
```

**Успешный ответ (`200`):**

```json
{ "ref_key": "c4e6092a-6cd2-11f1-bbb7-08f97e612452" }
```

`ref_key` — UUID созданного и **уже проведённого** заказа.

**Что изменилось в логике сайта:**
- ❌ Убрать отдельные POST партнёра (3.1), контрагента (3.2) и проведения (3.4).
- ❌ Не слать служебную «обвязку» шапки (`Организация_Key`, `Соглашение_Key`,
  `Валюта_Key`, `Склад_Key`, `СтавкаНДС_Key`, `ВидЦены_Key`, `ЭтапыГрафикаОплаты`,
  и т.п.) — все константы 1С проставляет сама (раздел 4.5 ТЗ зашит в 1С).
- ✅ Слать только бизнес-данные: покупатель, номер, сумма, товары.
- ✅ Проведение уже включено — повторный запрос на `/Post()` не нужен.
- ⚠️ Заказ создаётся со статусом «К выполнению / В резерве» (типовой статус заказа
  со склада); это обеспечивает корректное отображение состояния и попадание в
  `orders-in-shipping`.

**Обработка ошибок POST:**

| HTTP | code | Причина |
|---|---|---|
| 400 | `MISSING_ITEMS` | `Товары` пуст или отсутствует |
| 400 | `MISSING_BUYER` | `Покупатель` пуст |
| 404 | `ITEM_NOT_FOUND` | UUID товара не найден (в `error` — номер строки) |
| 500 | `INTERNAL_ERROR` | Ошибка проведения/записи (текст в `error`) |

## 5. Поля, которые ИСЧЕЗЛИ (и почему это безопасно)

| Поле ТЗ | Где было | Почему убрано |
|---|---|---|
| `Period` | Цены (2.5) | 1С отдаёт срез последних — последняя цена уже выбрана на сервере |
| `RecordType` (`Receipt`/`Expense`) | Остатки (2.6) | 1С отдаёт готовый рассчитанный остаток (`in_stock`), сайту не нужно суммировать движения |
| `Состояние` | Состояния заказов (2.7) | Фильтр по статусу выполняет 1С, в ответ попадают только нужные заказы |
| Ответы партнёра/контрагента (`Ref_Key`) | POST 3.1, 3.2 | Создаются внутри `/order`, наружу не отдаются — сайту нужен только `ref_key` заказа |

Все остальные поля ТЗ присутствуют (часть — переименована в snake_case,
см. разделы 3–4).
